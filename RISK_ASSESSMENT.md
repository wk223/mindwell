# 观心 MindWell — 风险评估报告

> 评估日期：2025-07-17 | 版本：v1.0

---

## 一、安全风险

| # | 风险项 | 严重度 | 现状 | 影响 | 建议措施 |
|---|--------|--------|------|------|---------|
| S1 | **JWT Token 存 localStorage** | 🔴 高 | Token 明文存储于 localStorage，XSS 攻击可直接窃取 | 攻击者获取 token 后可冒充用户，访问全部对话/日记/测评数据 | 改用 httpOnly cookie + CSRF token；短期至少加 token 加密 |
| S2 | **前端无 CSP 头** | 🟡 中 | Nginx 未配置 Content-Security-Policy | XSS 攻击面扩大，恶意脚本可注入 | Nginx 添加 CSP 头，限制 script-src 和 connect-src |
| S3 | **LLM API Key 硬编码风险** | 🟡 中 | `.env` 中存储，未加密，compose 通过环境变量明文传入容器 | 服务器被入侵后 API key 泄露，产生费用 + 数据泄露 | 使用 Docker secrets 或 HashiCorp Vault 管理密钥 |
| S4 | **CORS 生产环境为 `*`** | 🟡 中 | `docker-compose.prod.yml` 中 `CORS_ORIGINS=["*"]` | 任意域名可调用 API（但需要 token），增大 CSRF 风险 | 限制为具体域名列表 |
| S5 | **密码强度无要求** | 🟢 低 | 注册仅要求 ≥8 字符，无复杂度校验 | 弱密码可被暴力破解 | 前端 + 后端双重校验（大小写+数字+特殊字符） |
| S6 | **无速率限制差异化** | 🟢 低 | 全局限流已实现，但登录/注册接口未单独严格限流 | 暴力破解登录风险 | 登录接口单独限流（5次/分钟/IP） |

## 二、技术风险

| # | 风险项 | 严重度 | 现状 | 影响 | 建议措施 |
|---|--------|--------|------|------|---------|
| T1 | **无自动化测试** | 🔴 高 | 项目零单元测试/集成测试/E2E | 每次修改可能引入回归 bug，无法验证核心流程 | 优先补：auth API 测试 + dialogue 流式测试 + 前端关键路径 E2E |
| T2 | **Three.js 打包体积** | 🟡 中 | three+r3f+drei≈193KB gzip，React.lazy 分割 | 低端设备/弱网首次加载情绪星球页面 2-4 秒 | 保持 lazy load，加 loading 骨架屏，dpr 限制 |
| T3 | **无前端错误监控** | 🟡 中 | 仅 ErrorBoundary 捕获渲染错误，生产环境用户端 JS 报错不可见 | 线上 bug 无法及时发现和定位 | 接入 Sentry 或自建错误收集 |
| T4 | **数据库无备份策略** | 🟡 中 | compose 中 backup 容器仅示例，未配置定时备份 | 数据丢失不可恢复（用户对话/日记/测评全部丢失） | 配置 pg_dump 定时任务 + 异地备份 |
| T5 | **PostgreSQL RLS 依赖 SET 命令** | 🟡 中 | `dependencies.py` 用 f-string 拼 SQL 设置 `app.current_user_id` | 已修复参数绑定问题，但 f-string 拼 SQL 仍有注入风险（当前 UUID 来自数据库可信） | 改用 PostgreSQL `set_config()` 函数或 session 变量 |
| T6 | **LLM 单点依赖** | 🟡 中 | 仅依赖 DeepSeek API，无 fallback | DeepSeek 宕机 → AI 对话全部不可用 | 增加 Ollama 本地模型作为 fallback，LLM client 增加重试+降级逻辑 |
| T7 | **前端状态管理碎片化** | 🟢 低 | 8 个 Zustand store 独立，部分 store 间存在隐式依赖 | 跨 store 状态不一致风险 | 文档化 store 依赖关系，必要时合并 |

## 三、运维风险

| # | 风险项 | 严重度 | 现状 | 影响 | 建议措施 |
|---|--------|--------|------|------|---------|
| O1 | **单服务器无高可用** | 🔴 高 | 全部服务跑在同一台腾讯云服务器 | 服务器宕机 → 全部服务不可用 | 短期：加健康检查 + 自动重启；长期：多节点 + 负载均衡 |
| O2 | **无日志聚合/告警** | 🟡 中 | Docker logs 仅本地存储，无 ELK/Loki | 故障排查困难，无法回溯历史问题 | 接入 Loki + Grafana 或至少配置 Docker log driver 输出到文件 |
| O3 | **Redis 无持久化** | 🟡 中 | Redis 仅内存模式，重启后数据丢失 | 速率限制计数重置，对用户体验影响较小 | 开启 AOF 持久化 |
| O4 | **SSL 证书手动管理** | 🟢 低 | Certbot + docker-compose.ssl.yml 自动续期 | 证书过期 → HTTPS 不可用 | 当前方案已覆盖，保持监控 |
| O5 | **Ollama 与 DeepSeek 混用混乱** | 🟢 低 | compose 配置曾硬编码 ollama 覆盖 .env 导致 API 500 | 已修复，但需文档明确何时用 Ollama 何时用 API | 编写部署文档，明确 LLM 配置方式 |

## 四、产品/体验风险

| # | 风险项 | 严重度 | 现状 | 影响 | 建议措施 |
|---|--------|--------|------|------|---------|
| P1 | **AI 回复质量不稳定** | 🟡 中 | 依赖 DeepSeek API 版本，无 A/B 测试 | 用户对 AI 回复不满 → 流失 | 增加用户反馈机制（👍👎），收集数据优化 prompt |
| P2 | **危机检测误判/漏判** | 🔴 高 | Safety Pipeline 基于关键词+规则引擎，无 ML 模型 | 误判：正常用户被打断体验；漏判：真正危机未触发引导 | 引入轻量情感分类模型，降低误报率 |
| P3 | **日间模式视觉接受度** | 🟢 低 | 刚完成暖绿调日间改造，未经过用户测试 | 部分用户可能不习惯绿色调 | 增加用户自定义主题色选项 |
| P4 | **3D 情绪星球性能** | 🟡 中 | 2000 星场粒子 + 8 颗 48 段球体，移动端可能掉帧 | 移动端用户无法使用情绪星球 | 移动端检测 → 自动降级为静态 2D 展示 |
